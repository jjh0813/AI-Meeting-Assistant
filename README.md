# 회의록 AI — 녹음 파일 분석 & 부서별 요약·일정·보고서 자동화

> 녹음 파일을 올리면 **받아쓰기(STT) → 사람 검토 → 개인정보 마스킹 → DB 저장 → 요약·일정 추출 → 보고서 자동 작성**까지 해주는 웹 서비스.
> 회계·경영·영업 3개 부서의 데이터가 **서로 절대 침범하지 못하게** 권한을 나누는 것이 핵심.

이 문서는 **하나씩 정하고 채워나가는 TODO 방식**으로 작성합니다.
지금은 요구사항·기술 스택·구조까지만 정리한 **초안**이고, 단계별 상세 TODO는 진행하면서 함께 추가합니다.

## 📋 요구사항 정리

### 왜 만드는가
- 녹음을 사람이 직접 듣고 정리하는 데 시간이 많이 든다.
- 일정·담당자·요청사항·결정사항이 누락되거나 잘못 전달될 수 있다.
- 부서별로 필요한 정보만 자동 정리·공유되는 방식이 필요하다.

### 반드시 지켜야 할 것
1. **사용자 검토** — STT(받아쓰기) 결과가 실제 녹음과 맞는지 사람이 확인·수정할 수 있어야 한다.
2. **부서 완전 분리** — 회계 / 경영 / 영업의 데이터·AI가 서로 침범하면 안 된다.
   - 잘못 노출되면 회사 입장에서 **내부 기밀 위반**으로 본다.
   - LLM 단계가 아니라 **그 전 단계인 DB부터** 권한 관리가 들어가야 한다. (텍스트 → DB → LLM 순서이기 때문)
3. **개인정보 마스킹** — 이름·전화번호 같은 민감정보는 마스킹된 채로 저장·출력된다.
   - 녹음이 **DB에 들어갈 때부터** 마스킹된 상태로 저장.
   - **① 마스킹된 본문**과 **② 개인정보 원본**을 분리해서 따로 저장.
   - 개인정보는 **해당 부서만** 열람 가능. 관리자도 **자기 부서 것만** 볼 수 있다.
4. **일정 권한** — 일정은 **담당자만** 접근할 수 있다.
5. **보고서 형식 결과물** — 담당자가 만든 **보고서 템플릿의 빈칸을 AI가 채우는** 방식. 텍스트 또는 엑셀로 출력.
6. **DB는 PostgreSQL**, RAG(문서 검색)는 **pgvector** 사용.
7. 비용 문제로 부서마다 AI를 따로 두지 않고, **하나의 AI를 공유하되 범위만 나눠서** 사용.

## 🛠 기술 스택 (Tech Stack)

> "기술 스택"은 요리로 치면 **주방에 갖춰둔 도구·재료 세트**입니다. 어떤 도구로 어떤 일을 할지 미리 정해두는 것.

| 구분 | 선택 | 이유 |
| --- | --- | --- |
| 백엔드 | **FastAPI** | 가볍고 빠른 파이썬 웹 프레임워크. 처리 흐름이 일직선이라 별도 에이전트 프레임워크 없이 **함수 파이프라인**으로 충분. |
| STT (받아쓰기) | **CLOVA Speech** (네이버) | 한국어 인식 정확도가 높고, 서버로 쓸 NCP와 같은 생태계. |
| LLM | **Gemma (로컬 실행, Ollama)** | 요약·정보 추출·보고서 작성용. 데이터가 외부로 나가지 않아 **기밀 유지에 유리**하고 토큰 요금이 없다. **교체 가능한 인터페이스**로 설계해, 나중에 다른 모델/외부 API로 바꿔도 코드를 거의 안 고쳐도 되게 한다. |
| DB | **PostgreSQL + pgvector** | 권한 관리와 RAG 벡터 검색을 같은 DB에서 처리. |
| 웹 서버(배포 시) | **nginx** | 배포 단계에서 FastAPI 앞에 두어 HTTPS·보안·요청 전달 담당. 개발 중에는 없어도 됨. |
| 서버 | **Naver Cloud Platform (NCP)** | 배포 대상. (관리형 DB / VM+Docker 중 택일 — 배포 단계에서 확정) |
| 패키지 관리 | **uv** | 빠른 파이썬 패키지·가상환경 매니저. |
| 배포 방식 | **서버 직접 설치** | NCP 서버에 SSH로 접속해 nginx·Python·PostgreSQL을 직접 설치·운영한다. (Docker·CI/CD는 지금은 쓰지 않음 — 필요해지면 나중에 추가) |

### 아키텍처 흐름 (계층 구조)

셋은 경쟁 관계가 아니라 **역할이 다른 층**입니다.

```
[클라이언트] → (배포 시) nginx → FastAPI → Gemma(로컬, Ollama)
                          │         │            │
                       전달·보안   모든 판단     텍스트
                       HTTPS      (권한·마스킹   요약·추출만
                                  ·DB·검토 등)
```

- **nginx** = 정문 안내데스크. 요청을 전달·암호화만 함(로직 없음). **개발 중엔 생략, 배포 때 추가.**
- **FastAPI** = 실제 직원. 로그인·부서 권한·마스킹·DB 분리 저장·검토·보고서 등 **모든 요구사항 로직이 여기 산다.** 그래서 반드시 필요.
- **Gemma** = 글 쓰는 전문가. 텍스트 요약·추출만 함. 권한·DB는 모름. FastAPI가 "교체 가능한 인터페이스"로 호출.

## 📁 프로젝트 구조 (목표 설계 — 진행하며 수정)

> 아직 코드는 없습니다. 아래는 "이렇게 만들 계획"이라는 지도입니다.
> 핵심은 **처리 단계별로 폴더를 나눈 것**(`services/`)과 **DB 접근을 권한 계층으로 감싼 것**(`repositories/`)입니다.

```
ai_meeting_project/
├── app/                        # 애플리케이션 코드
│   ├── main.py                 # FastAPI 진입점 (앱 생성·라우터 등록)
│   ├── core/
│   │   ├── config.py           # 환경변수 설정
│   │   └── prompts.py          # LLM 프롬프트 모음 (요약·추출·보고서)
│   ├── schemas/                # API 요청/응답 데이터 모양 정의
│   ├── api/routes/             # 바깥에서 들어오는 요청 창구(엔드포인트)
│   │   ├── upload.py           #   녹음 업로드 → STT
│   │   ├── review.py           #   STT 결과 사람이 검토·수정
│   │   ├── report.py           #   보고서 생성·다운로드 (텍스트/엑셀)
│   │   └── auth.py             #   로그인·부서별 권한
│   ├── services/               # 핵심 처리 로직 (파이프라인)
│   │   ├── stt.py              #   CLOVA Speech 연동
│   │   ├── masking.py          #   개인정보 탐지·마스킹
│   │   ├── analyzer.py         #   LLM 요약 + 일정/담당자/기한 추출
│   │   └── report_filler.py    #   보고서 템플릿 빈칸 채우기
│   └── repositories/           # DB 접근 계층 (부서별 권한 필터 포함)
│       ├── transcript.py       #   마스킹된 본문 저장·조회
│       ├── pii.py              #   개인정보 원본 (분리 저장·부서별 제한)
│       ├── schedule.py         #   일정 (담당자만 접근)
│       └── rag.py              #   pgvector 문서 검색
├── data/
│   ├── schema.sql              # 테이블·pgvector·권한 정의
│   └── templates/              # 보고서 템플릿 (담당자 제공)
├── tests/                      # pytest 테스트 (직접 실행)
├── .env.example                # 환경변수 템플릿
└── pyproject.toml
```
